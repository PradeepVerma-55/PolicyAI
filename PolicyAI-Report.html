PolicyAI is a production-grade Retrieval-Augmented Generation (RAG) system that transforms static policy documents into an interactive, always-cited knowledge assistant. It answers employee questions across HR, IT Security, Remote Work, Security Training, and Incident Response — grounded entirely in official documents, with zero hallucination.

The system indexes 1,602 chunks from five publicly available policy documents into a local Qdrant vector database. Every query performs semantic search across all sources simultaneously, retrieves the top-6 most relevant chunks, and generates a grounded answer through Groq's llama-3.1-8b-instant model — including the exact policy name, category, and page number for each source cited.

Key design principle: The LLM is strictly constrained to answer only from retrieved context. If the answer is not in the indexed documents, the system says so. This eliminates hallucination by design, not by luck.

02 — Problem Statement

The Problem

Every organisation accumulates hundreds of policy documents across HR, IT Security, compliance, and operations. These documents share a common fate: they are written once, stored somewhere, and almost never consulted again — not because they are unimportant, but because they are unsearchable.

⌛ Time Cost

A typical policy question takes 25–40 minutes to answer manually — opening PDFs, searching, cross-referencing versions, and replying. At 500 employees asking 3 questions per week, that's 1,500 manual searches every week.

⚠ Accuracy Risk

Manual searches produce inconsistent answers depending on who responds. Employees often receive outdated information, the wrong document version, or a best-guess answer that no one can trace back to a specific policy clause.

05 — Core Concept

RAG Pipeline Deep Dive

What is RAG?

Retrieval-Augmented Generation augments a language model with a retrieval step. Instead of relying on the model's parametric memory (what it learned during training), RAG fetches relevant evidence from an external knowledge source at inference time, then instructs the LLM to answer strictly from that evidence.

📊 Semantic Search vs Keyword Search

Traditional search matches exact words. Semantic search matches meaning. The query "What do I do if someone hacks us?" retrieves the incident response guide even though the document never uses the word "hacks" — because the 384-dimensional vector captures conceptual similarity, not string overlap.

🎯 Why 500 Characters Per Chunk?

Chunks must be small enough to be topically focused (so retrieval is precise) but large enough to be self-contained (so the LLM has enough context). 500 chars is ~80–100 words — roughly one policy clause or procedure step. The 50-char overlap prevents answers from being cut across boundaries.

Metadata Stamping — Why It Matters

Every chunk carries policy_name and policy_category in its metadata, stamped at load time before splitting:

# Before splitting — stamp every page
for doc in docs:
    doc.metadata["policy_name"]     = source["name"]
    doc.metadata["policy_category"] = source["category"]

# After splitting — every child chunk inherits the metadata
chunks = splitter.split_documents(docs)
# chunk.metadata = {"source": "...", "page": 3,
#                   "policy_name": "NIST SP 800-61r2: ...",
#                   "policy_category": "Incident Response"}

This is why every source citation in the UI shows not just the filename and page, but the human-readable policy name and domain category — without any post-processing.

06 — LangChain

The LCEL Chain

LangChain Expression Language (LCEL) lets you declare the entire RAG pipeline as a single composable expression using the | operator. Each | passes the output of the left component as input to the right.

rag_chain = (
    {
        "context":  retriever | format_docs,   # branch 1: retrieve → format
        "question": RunnablePassthrough(),      # branch 2: pass question unchanged
    }
    | RAG_PROMPT          # fill {context} and {question} slots
    | llm                 # ChatGroq → returns AIMessage
    | StrOutputParser()   # extract .content string
)

Step-by-Step for "What do I do after a security breach?"

StepComponentInputOutput
1 retriever "What do I do after a security breach?" List of 6 Document objects from Qdrant
2 format_docs List[Document] Cited context string with [Policy | Category | Page] labels
3 RunnablePassthrough The question string The question string, unchanged
4 RAG_PROMPT {context: "...", question: "..."} Filled ChatPromptValue (system + human messages)
5 ChatGroq ChatPromptValue AIMessage with generated answer
6 StrOutputParser AIMessage Plain string answer

Swapping providers is one line. Because ChatGroq, ChatOpenAI, and ChatAnthropic all implement the same ChatModel interface, changing the LLM provider requires editing exactly one line — nothing else in the chain needs to know.

07 — Implementation

Vanilla Python vs LangChain

The project implements the same RAG pipeline in two ways. The comparison makes concrete what the framework actually abstracts away.

Step🐍 vanilla_rag🦀 langchain_rag
Load PDF requests.get() + fitz.open() + manual page loop PyMuPDFLoader(path).load() → List[Document]
Chunking Custom 100-word sliding window, no boundary awareness RecursiveCharacterTextSplitter — splits at ¶¶ → ¶ → space → chars
Embed SentenceTransformer.encode(texts) — raw numpy arrays HuggingFaceEmbeddings.embed_documents() — LangChain interface
Store Manual PointStruct(id, vector, payload) + client.upsert() QdrantVectorStore.from_documents(docs, embedder) — one call
Retrieve client.query_points(collection, query_vector, limit) vectorstore.as_retriever(k=6).invoke(query)
Call LLM groq.chat.completions.create(model, messages) ChatGroq(model, api_key) — same interface as OpenAI/Anthropic
Parse Output response.choices[0].message.content StrOutputParser() extracts .content from AIMessage
Full Pipeline Explicit step-by-step rag() function LCEL: retriever | format_docs | prompt | llm | parser
Swap LLM Edit multiple lines across the function Change one line: ChatGroq → ChatOpenAI

09 — Examples

Sample Q&A

Q What do I do if I suspect a phishing attack?
When a security incident is suspected, users should immediately report it to the IT Security team. Preserve all evidence — do not delete emails or logs. Isolate the affected system from the network if possible. Document the timeline of events and actions taken. Do not attempt remediation without guidance from the security team.
Source: NIST SP 800-61r2: Computer Security Incident Handling [Incident Response] — Page 21
Q What security measures are required for remote work?
Remote workers must use organisation-approved VPN for all access to internal resources. Devices must have up-to-date antivirus software, full-disk encryption enabled, and automatic lock after 5 minutes of inactivity. Sensitive data must not be stored on personal devices or cloud services not approved by IT.
Source: NIST SP 800-114r1: Telework & Remote Access Security [Remote Work Security] — Page 12
Q What are the leave entitlements for employees?
Employees are entitled to 18 Earned Leaves per calendar year. Leave is calculated on a pro-rata basis for mid-year joiners. Sick leave beyond 3 consecutive days requires a medical certificate. Unused Earned Leave may be carried forward up to a maximum of 45 days.
Source: HR Policy [Human Resources] — Page 3
